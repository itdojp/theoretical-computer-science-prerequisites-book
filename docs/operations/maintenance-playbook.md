# メンテナンス・プレイブック

## 目的

このページは、教材を継続運用するための定期点検、障害対応、改善サイクルを定めます。

## 定期点検

### 毎週

- 未対応Issueを確認する。
- 受講者からの質問を分類する。
- 誤答が集中した問題を記録する。
- リンクチェックとpytestを実行する。

### 毎月

- 章別レビュー問題の通過率を見る。
- 低通過率の章に例・非例・解説を追加する。
- 使われていないページを確認する。
- 新規PRの取り込み方針を決める。

### 四半期

- 本体教科書との対応表を見直す。
- 用語索引・記号索引の表記揺れを修正する。
- ライセンス・引用ポリシーを確認する。
- v1.0に向けた破壊的変更の有無を判断する。

## Issueトリアージ

| 種別 | 優先度 | 対応目安 |
|---|---:|---|
| 誤った定義・証明 | 高 | 即時修正 |
| 解答の誤り | 高 | 即時修正 |
| リンク切れ | 中 | 次回リリースまで |
| 説明不足 | 中 | 誤答データを見て判断 |
| 新章追加要望 | 低 | スコープに合うか審査 |
| 表記揺れ | 低 | まとめて修正 |

## CI失敗時の対応

### リンクチェック失敗

1. エラーに表示されたファイルを開く。
2. 相対パスが、リンク元ファイルから見て正しいか確認する。
3. ファイル名変更による破損なら、docs/_data/navigation.yml も更新する。
4. `python scripts/check_links.py` を再実行する。

### 演習ラベルチェック失敗

1. 対象演習ファイルを開く。
2. 難易度ラベル `[A]`〜`[E]` が付いているか確認する。
3. 冒頭の「難易度ラベル」説明が残っているか確認する。
4. `python scripts/check_exercise_labels.py` を再実行する。

### pytest失敗

1. 失敗したテスト名を確認する。
2. 実装の仕様変更か、テストの期待値誤りかを分ける。
3. 仕様変更の場合は、本文・Pythonノート・ミニプロジェクトも更新する。
4. `python -m pytest` を再実行する。

## 内容更新の判断基準

次の条件を満たす更新を優先します。

- 複数の受講者が同じ箇所で詰まった。
- 本体教科書の読解に直接影響する。
- 1つの定義・例・非例・演習で解消できる。
- 既存の章構成を壊さない。

逆に、次の更新は慎重に扱います。

- 高度な理論を前倒しで追加する。
- 特定の大学講義の章構成に寄せすぎる。
- 実装テクニックだけを増やす。
- AI・MLなど本教材の目的外の範囲を増やす。

## バックアップ

正式運用では、次を保存します。

- リリースZIP
- Gitタグ
- 受講者提出物
- 採点表
- レビュー会議事録
- リリースチェックリスト

受講者提出物と採点表には個人情報が含まれ得るため、教材リポジトリとは分離します。

## 変更記録

変更は `CHANGELOG.md` に記録します。最低限、次を残します。

- 追加したページ
- 修正した章
- 変更した演習
- 修正した誤り
- 実行した品質チェック
- 既知の制限
