# コントリビューションガイド

## 目的

このページは、本教材に修正・追加を行う際の基準を定めます。

この教材は、理論計算機科学本体の前提補強が目的です。範囲外の高度な話題を増やすより、読者が定義・証明・擬似コードで止まらないようにする修正を優先します。

## 受け入れる変更

| 種別 | 例 |
|---|---|
| 誤字修正 | 記号、用語、表記揺れの修正 |
| 説明改善 | 定義、例、非例、証明の補足 |
| 演習追加 | 前提補強に必要な短い問題 |
| 解答改善 | 証明のステップ追加、誤答例の説明 |
| 図表追加 | 証明構造、グラフ、DFA、状態遷移など |
| 実装例追加 | 小さく、テスト可能なPython例 |
| 運用改善 | 採点、レビュー、進捗管理、CI |

## 原則

1. 本体教科書を置き換えない。
2. 前提補強に必要な範囲に絞る。
3. 定義、例、非例、演習、解答の対応を崩さない。
4. 外部教材の本文・図・問題を転載しない。
5. 追加したコードにはテストを付ける。
6. 新規Markdownには相対リンクを使う。
7. 演習には難易度ラベル `[A]`〜`[E]` を付ける。

## Pull Request の条件

PRには次を含めます。

- 変更目的
- 変更対象ファイル
- 前提補強として必要な理由
- 追加・変更した演習の解答
- 実行したチェックコマンド
- 既知の未解決点

テンプレートは `.github/PULL_REQUEST_TEMPLATE.md` にあります。

## 文章スタイル

| 項目 | 方針 |
|---|---|
| 文体 | です・ます調ではなく、説明文として簡潔に書く |
| 用語 | 同じ概念には同じ訳語を使う |
| 記号 | 既存の記号索引と揃える |
| 例 | 小さい有限集合・短い文字列・小さいグラフを優先する |
| 非例 | 読者が混同しやすいものを選ぶ |
| 証明 | 仮定、示すこと、定義展開、結論を明確にする |

## 演習追加ルール

演習を追加する場合は、次を満たします。

```text
- 問題文が自己完結している。
- 難易度ラベルが付いている。
- 解答または解答方針がある。
- 対応章が明確である。
- どの本体章の前提に効くかが説明できる。
```

難易度の基準は [演習難易度インデックス](../exercises/difficulty-index.md) を参照します。

## 図表追加ルール

図表は、見た目より概念対応を優先します。

- SVGを推奨する。
- ファイル名は英小文字とハイフンにする。
- 本文中で図の読み方を説明する。
- 図だけで新しい定義を導入しない。

## コード追加ルール

Python examples を追加する場合は、次を満たします。

- 標準ライブラリ中心にする。
- 主要関数に型ヒントを付ける。
- 例外条件を明示する。
- `tests/` または `examples/python/tests.py` にテストを追加する。
- ランダム性がある場合はseedを固定できるようにする。

## レビュー観点

レビュー担当者は、次を確認します。

- 教材のスコープを超えていないか。
- 定義が正確か。
- 例と非例が適切か。
- 証明に飛躍がないか。
- 演習と解答が対応しているか。
- 新規リンクが存在するか。
- CIが通っているか。
- ライセンス上危険な転載がないか。
